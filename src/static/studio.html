Справка Data Studio

Обзор — дашборд системы

Сводная панель: источники данных и их статусы, поддерживаемые темы, основные KPI города. Быстрый доступ к тестированию и управлению городами.

API — документация эндпоинтов

Встроенный Swagger UI с интерактивным тестированием всех API-эндпоинтов. Поддерживает Try It Out для мгновенной проверки запросов.

Тестирование — pytest + загрузка данных

Запуск автоматических тестов (pytest) и массовая загрузка данных из всех источников. Показывает статус каждого датасета (загружен / устарел / отсутствует).

Города — городские профили

Каждый профиль описывает один город: ID, название, временная зона, список районов, доступные датасеты. Профили хранятся в config/city_profile_*.yaml. Нажмите «Активировать» чтобы переключить бота на другой город — изменение применяется без перезапуска. «Новый профиль» создаёт заготовку YAML — заполните поля и нажмите «Сохранить».

📥 Импорт данных — загрузка статических датасетов

Загружает файлы со статическими данными (тепловые источники, выбросы, метро и др.) в формате CSV, JSON или GeoJSON. Мастер из 4 шагов: выбор города и типа → загрузка файла → маппинг колонок → результат.

CSV — кодировка UTF-8 или Windows-1251, разделитель «;» или «,». Первая строка — заголовки. Координаты ожидаются в колонках lat / lon или широта / долгота.
JSON — плоский массив объектов [{"{"}…{"}"}, …] или объект с ключом data/rows/features.
GeoJSON — стандартный FeatureCollection. Атрибуты из properties, координаты из geometry.coordinates.
🔗 Онлайн источники — URL обновления данных

Каждый онлайн-датасет привязан к URL CSV-файла на opendata.novo-sibirsk.ru (или аналоге для другого города). TTL обновления — 24 ч по умолчанию. Изменение URL вступает в силу при следующем запросе POST /update.

📋 Справочник схем — канонические поля

Показывает обязательные и опциональные поля для каждого типа датасета. Имена полей — контракт между бэкендом и картой: их нельзя переименовывать без одновременного изменения кода в src/. При маппинге импорта ориентируйтесь именно на этот справочник.

⚙️ Регламенты — коэффициенты бизнес-логики

Позволяет изменять пороги и коэффициенты расчётных индексов (пробки, экология, индексы жизни) без перезапуска сервера. YAML-редактор + быстрые слайдеры для ключевых параметров. Кнопка «Применить» сохраняет файл и мгновенно перезагружает логику. Подробнее — см. инструкцию аналитика внутри вкладки.

Инструкция аналитика регламентов
Что такое регламент

Каждый регламент — YAML-файл в config/rules/. В них хранятся все пороговые значения, коэффициенты и тексты рекомендаций. Изменение файла через эту панель применяется мгновенно без перезапуска сервера.

Четыре файла
  • 🚦 traffic_rules.yaml — индекс пробок: часы пик, погодные Δ, тексты советов
  • 📊 life_indices_rules.yaml — индексы водителя, прогулки, коммунальной нагрузки
  • 🌿 ecology_rules.yaml — пороги рисков PM2.5, гололёд, температурный шок
  • 📅 holiday_calendar.yaml — производственный календарь 2025–2027
Рабочий процесс
  1. Выберите регламент в левой панели
  2. Отредактируйте YAML в текстовом редакторе или передвиньте слайдер (только traffic_rules)
  3. Нажмите ✓ Применить — файл сохранится и логика перезагрузится
  4. Проверьте результат в главном интерфейсе
Безопасность изменений
  • ⬇ Экспортируйте YAML перед крупными правками — это ваш бэкап
  • ↺ Сбросить откатывает редактор к последнему сохранённому состоянию (файл не перезаписывается)
  • Если после «Применить» логика работает неверно — восстановите файл из экспорта через тот же редактор
  • YAML чувствителен к отступам — используйте пробелы, не табы
Структура YAML: типы значений
weekend_scale: 0.45 # число
label: "Свободно" # строка
morning_only: true # булево
codes: [71, 73, 75] # список
Советы для traffic_rules.yaml
  • weekend_scale: 0.3–0.6 — чем ниже, тем тише выходной день
  • Δ погоды: +1.0–+3.0 — разумный диапазон для одного фактора
  • tips: используйте {"{transport_hint}"} — автоматически заменяется на сезонный транспорт
  • seasonal.scooter: формат дат MM-DD (напр. 05-01)
